## 4.6 网关运维

Gateway 作为长期运行的服务，需要正确的部署和运维策略。本节覆盖服务管理、日志监控和常见问题排查。

---

### 4.6.1 `hermes gateway` 命令

Gateway 的所有运维操作都通过 `hermes gateway` 子命令管理：

```bash
# 前台运行（调试用，Ctrl+C 停止）
hermes gateway run

# 后台服务管理
hermes gateway start    # 启动为系统服务（systemd/launchd）
hermes gateway stop     # 停止服务
hermes gateway status   # 检查服务状态
hermes gateway restart  # 重启服务
```

`hermes gateway start` 会根据操作系统自动选择服务管理器：
- **Linux**：创建 systemd user service
- **macOS**：创建 launchd LaunchAgent

---

### 4.6.2 Linux systemd 服务

手动创建或由 `hermes gateway start` 自动生成：

```ini
# ~/.config/systemd/user/hermes-gateway.service
[Unit]
Description=Hermes Agent Gateway
After=network-online.target

[Service]
ExecStart=/path/to/hermes gateway run
Restart=on-failure
RestartSec=10

[Install]
WantedBy=default.target
```

如果使用 Profile（多配置），服务名会带上 profile 后缀：

```
hermes-gateway.service              # 默认配置
hermes-gateway-work.service         # --profile work
hermes-gateway-personal.service     # --profile personal
```

常用操作：

```bash
# 查看日志
journalctl --user -u hermes-gateway -f

# 重启
systemctl --user restart hermes-gateway

# 开机自启
systemctl --user enable hermes-gateway
```

---

### 4.6.3 macOS launchd 服务

```xml
<!-- ~/Library/LaunchAgents/com.nousresearch.hermes-gateway.plist -->
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.nousresearch.hermes-gateway</string>
    <key>ProgramArguments</key>
    <array>
        <string>/path/to/hermes</string>
        <string>gateway</string>
        <string>run</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
</dict>
</plist>
```

```bash
# 加载服务
launchctl load ~/Library/LaunchAgents/com.nousresearch.hermes-gateway.plist

# 卸载
launchctl unload ~/Library/LaunchAgents/com.nousresearch.hermes-gateway.plist
```

---

### 4.6.4 `/restart` 远程重启

在任何已连接的平台上发送 `/restart` 命令，可以远程重启网关：

```
/restart
```

Agent 会发送确认消息后重启 Gateway 进程。适用于：
- 修改了 `config.yaml` 后需要生效
- Agent 行为异常需要重启
- 升级后重启加载新版本

---

### 4.6.5 更新时自动重启

运行 `hermes update` 升级 Hermes 时，会自动：

1. 发现所有运行中的 Gateway 服务（systemd/launchd）
2. 重启这些服务
3. 停止非服务模式的手动 `gateway run` 进程

```bash
hermes update
# 输出示例：
# ✓ Updated to v0.10.1
# ✓ Restarting hermes-gateway.service...
# ✓ Gateway restarted successfully
```

### 4.6.6 日志监控

Gateway 的日志输出到标准输出（stdout），systemd 环境下通过 `journalctl` 查看：

```bash
# 实时跟踪日志
journalctl --user -u hermes-gateway -f

# 查看最近 100 行
journalctl --user -u hermes-gateway -n 100

# 按时间过滤
journalctl --user -u hermes-gateway --since "1 hour ago"

# 搜索错误
journalctl --user -u hermes-gateway | grep -i error
```

#### 关键日志事件

| 日志关键词 | 含义 | 建议操作 |
|-----------|------|---------|
| `gateway:startup` | 网关启动完成 | 正常，无需操作 |
| `platform connected` | 平台连接成功 | 正常 |
| `connection lost` / `reconnect` | 平台连接断开重连 | 偶尔正常；频繁则检查网络 |
| `session expired` | 会话自动过期重置 | 正常，按 reset_policy |
| `rate limit` | 平台 API 限流 | 降低消息频率 |
| `token expired` | API Token 过期 | 重新生成 Token |
| `permission denied` | 权限不足 | 检查 Bot 权限配置 |

### 4.6.7 性能优化（v0.10.0+）

#### Agent 缓存 LRU + 空闲 TTL

长期运行的 Gateway 会缓存 Agent 实例，但缓存无限制会导致内存泄漏。v0.10.0 引入了 LRU 淘汰 + 空闲 TTL：

```yaml
# config.yaml
gateway:
  agent_cache:
    max_size: 50          # 最多缓存 50 个 Agent 实例
    idle_ttl: 3600        # 空闲 1 小时后淘汰
```

#### Session 状态自动清理

`maybe_auto_prune_and_vacuum()` 在启动时幂等执行，清理过期 session 和 FTS5 索引碎片。一个重度用户报告：384MB/982 sessions → prune + VACUUM 后降到 43MB。

#### 临时 Agent 自动关闭

一次性任务完成后，临时创建的 Agent 实例会自动关闭，防止资源泄漏。

### 4.6.8 常见问题排查

| 症状 | 可能原因 | 排查步骤 |
|------|---------|---------|
| 网关启动后立即退出 | Token 配置错误 | `hermes gateway run` 前台运行看错误信息 |
| 部分平台不响应 | 平台配置缺失或错误 | `/platforms` 检查连接状态 |
| 消息延迟严重 | 网络问题或 Token 限流 | 查看日志中的 `rate limit` 关键词 |
| 内存持续增长 | Agent 缓存泄漏 | 升级到 v0.10.0+，确认 LRU + TTL 生效 |
| Session 数据过大 | 长期未清理 | 检查 `reset_policy` 配置；手动重启触发 prune |
| 更新后行为异常 | 旧缓存/进程残留 | `hermes gateway stop` → `hermes gateway start` |
| Docker 中网关无法连接主机 | Docker 网络隔离 | 使用 `host.docker.internal` 或 `--network host` |

#### 全局隧道/代理场景

如果你的服务器通过 TUN 代理（Clash/Mihomo/Sing-box）、企业 VPN 或 Tailscale 访问外网，可能会遇到"请求被拦截但返回的是代理 IP"的问题。Hermes 默认阻止私有 IP 范围以防止 SSRF，需要手动放行：

```yaml
# config.yaml
security:
  allow_private_urls: true
```

或环境变量 `HERMES_ALLOW_PRIVATE_URLS=1`。

这会允许解析 198.18.0.0/15（TUN 代理）、100.64.0.0/10（Tailscale）等私有 IP 范围。云元数据端点（169.254.169.254 等）始终被阻断。
