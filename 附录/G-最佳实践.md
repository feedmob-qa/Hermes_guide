## 附录 G：最佳实践

### G.1 最小授权优先

Hermes 很强，但越强越需要克制。

建议顺序：

1. 先让 Agent 能回答问题
2. 再给只读工具：`web`、`vision`、`skills`
3. 再给文件读写：`file`
4. 最后才给 `terminal`、`browser`、`delegation`

团队和消息平台入口尤其如此。不要把 `hermes-cli` 那套全量工具直接照搬到公共群聊。

### G.2 按平台拆 Toolsets

推荐做法：

| 平台 | 推荐策略 |
|------|----------|
| CLI | 可相对开放，适合 `web`、`file`、`terminal`、`skills`、`todo` |
| Telegram 私聊 | 可在可控范围内开放，但建议保留审批 |
| Discord / Slack 团队频道 | 默认只开 `safe`、`web`、`vision`、`skills`、`memory` |
| Cron | 只保留任务真正需要的工具 |
| ACP / IDE 集成 | 优先保留代码相关工具，不要默认开放消息投递能力 |

### G.3 不要长期裸奔使用 `/yolo`

`/yolo` 或 `--yolo` 适合：

- 你在本地沙箱里做一次性实验
- 你明确知道本轮需要执行的命令范围
- 你愿意承担误操作风险

不适合：

- 团队共享机器人
- 面向真实业务数据的生产环境
- 开放群聊
- 带 root 权限或宿主机关键目录挂载的 Docker 环境

### G.4 Profile 分离

强烈建议至少拆成两套：

1. 个人环境：偏自由、偏开发
2. 工作环境：偏收敛、偏可审计

如果你既做开发又跑消息网关，再加第三套：

3. Bot / Gateway 环境：固定模型、固定工具集、固定 home channel

这样做的收益：

- 避免个人试验污染团队机器人
- 避免开发技能和生产网关配置混在一起
- 更容易备份、迁移和回滚

### G.5 本地开发优先用 Docker 后端

如果你希望 Agent 能写文件和跑命令，但又不想直接操作宿主机，优先上 Docker。

建议：

- 默认不开整盘挂载
- 只挂当前项目目录
- 容器里跑工作命令，宿主机只保存源代码和密钥
- 通过 `docker_forward_env` 精确转发少量变量

### G.6 群聊默认按用户隔离会话

`group_sessions_per_user: true` 通常应保持开启。

理由：

- 不同用户不会共享上下文
- 一个用户的错误操作不会污染整个群的上下文
- Token 成本更可控
- 中断、重试、压缩等行为更可预测

只有当你明确需要“整个群共享一个房间脑”时，才考虑关闭。

### G.7 记忆要短、小、稳定

好的记忆：

- 是稳定事实
- 能跨多次会话复用
- 不依赖瞬时状态

适合写进 `MEMORY.md` / `USER.md` 的例子：

- 用户偏好用中文回复
- 某仓库默认分支叫 `main`
- 团队统一用 `pnpm`
- 某项目部署在特定路径

不适合长期记忆的内容：

- 临时 bug 状态
- 今天下午要做什么
- 一次性的实验步骤

这类内容更适合放在当前会话、技能、Kanban 或 Cron 里。

### G.8 Skills 用来存流程，不要存废话

好的 Skill 应该：

- 解决重复出现的问题
- 有清晰触发条件
- 步骤可执行
- 明确依赖哪些工具或环境

不好的 Skill：

- 只是把长提示词随便存起来
- 复述常识，没有操作价值
- 绑定太多一次性上下文

### G.9 定时任务设计要轻量

Cron 最容易踩的坑是把“大任务”硬塞进“高频任务”。

建议：

- 每日摘要、每周审查、提醒类任务适合 Cron
- 大规模仓库扫描、长链路数据抓取不适合高频跑
- 大任务拆成脚本预处理 + Hermes 总结
- 多任务不要都压在整点

### G.10 压缩阈值要早点调

长对话最常见的问题不是“报错”，而是：

- 变慢
- 变贵
- 前文丢重点

建议：

- 默认保留自动压缩
- 本地小模型可以把 `threshold` 设得更低一些
- 最近消息保护不要无限上调
- 用 `/compress` 主动收束超长任务

### G.11 备份要常态化

至少建立两种备份习惯：

1. 版本升级前：`hermes backup --quick`
2. 结构调整前：`hermes backup`

建议备份对象：

- `config.yaml`
- `.env`
- `auth.json`
- `skills/`
- `memories/`
- `sessions/`
- `cron/`
- `kanban/`

### G.12 对外求助前先做“最小可复现”

出问题时先回答这几个问题：

1. 是 CLI、Gateway、Cron、Kanban 还是 MCP 场景？
2. 当前 provider/model 是什么？
3. 当前 terminal backend 是什么？
4. 当前平台 toolset 是什么？
5. 问题稳定复现还是偶发？
6. `hermes doctor` 和日志说了什么？

然后再把问题发给同事或社区。这样效率会高很多。

### G.13 团队使用时要有运营边界

如果 Hermes 在团队里长期运行，最好明确这些边界：

- 谁能修改 `config.yaml`
- 谁能改 `.env`
- 谁能启用 `/yolo`
- 哪些平台允许发命令，哪些平台只允许问答
- 哪些仓库目录允许写入
- 何时允许更新 Hermes 版本

没有边界时，问题会以“大家都能改，所以谁都说不清”的形式出现。

### G.14 先用官方向导，再做手工精修

推荐顺序：

1. `hermes setup`
2. `hermes model`
3. `hermes tools`
4. `hermes doctor`
5. 手工编辑 `config.yaml`

这样做的好处是：

- 先跑通最小路径
- 降低拼错键名的概率
- 让官方迁移逻辑先把旧配置修平

### G.15 文档维护建议

对于本仓库这种教程型项目，附录内容最容易过期。建议维护方式：

1. 优先写“结构”和“原则”，少写短周期细节
2. 可变清单尽量来自官方文档再整理
3. 每次升级 Hermes 版本后，先审附录中的命令、toolsets、provider 名称
4. 把“官方文档优先”写进仓库约定

### G.16 最后一条

Hermes Agent 不是一个“装上就永远不用管”的脚本工具，它更像一套长期运行的 Agent 基础设施。你给它多少权限、多少上下文、多少入口，它就会在多大范围内影响你的系统。配置质量，决定了它到底是助手，还是隐患。
