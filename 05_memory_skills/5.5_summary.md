## 5.5 本章小结

### 三种持久化机制

本章介绍了 Hermes 的三种持久化机制，它们互补而非竞争：

| 机制 | 存什么 | 容量 | 检索方式 | 适用场景 |
|------|--------|------|---------|---------|
| **Memory** | 稳定事实、偏好、环境信息 | ~3,500 字符 | 每轮自动注入系统提示 | "用户喜欢中文" |
| **Skills** | 程序化知识、工作流 | 无硬限制 | 渐进式披露（索引 → 完整加载） | "部署 ML 模型的步骤" |
| **Session Search** | 完整对话历史 | 无限（SQLite） | FTS5 全文搜索 + LLM 摘要 | "上次怎么配的 nginx" |

加上 **LMWiki 知识库**（结构化的领域知识）和 **用户偏好系统**（人格、Profiles），Hermes 的"记忆"远超单次对话的范畴。

### 关键设计决策

1. **冻结快照**：Memory 在会话启动时捕获快照注入系统提示，整个会话中不变。写入立即持久化到磁盘，但通过工具返回值让当前会话"看到"更新。这保护了 Anthropic Prefix Cache。

2. **渐进式披露**：Skills 不把完整指令塞进系统提示，而是只保留轻量索引。需要时才通过 `skill_view` 加载完整内容。系统提示节省了大量 token。

3. **策展而非堆积**：Memory 有严格的字符上限（MEMORY.md 2200 + USER.md 1375），空间不足时不会自动淘汰，而是让模型自行决定替换/删除什么。这保证了记忆的高质量。

4. **Profile 完全隔离**：每个 Profile 是独立的 HERMES_HOME，从配置、记忆、技能到 Gateway 全部隔离。适合不同职能角色的长期隔离。

5. **安全扫描无处不在**：Memory 写入、Skills 安装、SOUL.md 加载——所有入口都经过注入检测和威胁模式扫描。

### 自检清单

完成本章学习后，确认你能回答以下问题：

- [ ] Memory 的三层架构是什么？（存储层 MemoryStore / 编排层 MemoryManager / 插件层 MemoryProvider）
- [ ] 为什么 Memory 使用冻结快照而不是实时更新？（保护 Prefix Cache）
- [ ] 当 Memory 空间不足时会怎样？（返回错误，让模型自行策展）
- [ ] session_search 的两种模式是什么？（浏览最近 / 关键词搜索）
- [ ] 搜索时为什么要用 OR 连接关键词？（FTS5 默认 AND 会漏结果）
- [ ] Skills 的渐进式披露是什么意思？（索引在系统提示，完整内容按需加载）
- [ ] 如何让 Agent 自动创建 Skills？（设置 `creation_nudge_interval`）
- [ ] 什么时候用 Memory、什么时候用 Skills？（简单事实 → Memory，复杂流程 → Skill）
- [ ] SOUL.md 和 USER.md 的区别是什么？（SOUL 定义 Agent 身份，USER 描述用户画像）
- [ ] 如何创建一个独立的 Agent 角色？（创建 Profile + 配置 SOUL.md）
